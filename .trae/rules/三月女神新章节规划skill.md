# 三月女神新章节规划技能（轻量版）

## 定位

本规则只用于正常新章节写前校准、写前质量任务卡和已写章节复盘反推，不写正文，不做定稿入库，不直接重写跑偏章节。

## Master 角色定位

本 skill 在 Master 总导演调度制中承担：剧情规划师 + 连载节奏编辑 + 拆章协调员。

职责：
1. 判断本章事件是否成立、分场是否合理、商业连载意义是否足够。
2. 输出不可改事实、分场计划、质量任务卡和写前禁区。
3. 对已写章节做复盘反推，判断是微调质量卡、重做章纲，还是交给重写流程。
4. 执行人工裁决的拆章/合章操作（`/split-chapter`），同步章↔节拍映射、伏笔实际章号、正文文件和缓存；不自行决定拆分方案，只执行已确认的拆分。

不负责：
1. 不写正文，不做原位修文。
2. 不做定稿入库、references 同步或 Qdrant 上传（拆章时的 C 层入库清理由定稿入库 skill 执行）。
3. 不替代 `/review`、`/scene-check`、`/setting-term-check` 的专项检查。

被 Master 调用时，只处理章节规划和连载节奏维度，不自动扩展到其他角色职责。

重写跑偏章节请先用 `/chapter-review` 判断问题来源，再用 `/rewrite-plan` 制定重写方案。

## 推荐执行角色

使用擅长结构规划、设定校准和章节节奏拆分的规划型模型；不要依赖具体模型名。

## 必读文件

执行 `/chapter-plan` 时，优先读取：

1. `设定/大纲内容/第一卷/第一卷当前写作状态.md`
2. `设定/大纲内容/第一卷/第一卷写作设定包.md`
3. `设定/大纲内容/第一卷/第一卷口径卡片.md`
4. `设定/大纲内容/第一卷/第一卷事件链.md`
5. `设定/大纲内容/第一卷/第一卷章节目录版节奏表.md`
6. 当前章节所在细纲（若存在；第一卷第10章起不再创建章节细纲，若无细纲则跳过此项，直接从事件链和章节目录版节奏表派生分场计划）
7. 上一章正文定稿

**节拍制（2026-06-12 起）**：第一卷起，事件链/节奏表/分卷大纲等规划文件中的 ChN/第N章 一律指**节拍编号**，不是正文章号承诺（总约定见《第一卷事件链》第零节）。`/chapter-plan` 必须先读《当前写作状态》"一点五、章↔节拍映射表"确认已写到哪个节拍，再决定本章吃入的节拍范围（一章可吃一个节拍、一个半节拍，大节拍可拆两章）；分场计划从节拍的"必须发生/必须留下"项派生。一个节拍拆成多章写时，中间章的停点由本命令现场决定，**不需要修改任何规划文件**。

第 N 卷通用模板：第二卷及以后按当前卷目录读取同结构文件，即 `第N卷当前写作状态.md`、`第N卷写作设定包.md`、`第N卷口径卡片.md`、`第N卷事件链.md`、`第N卷章节目录版节奏表.md`、当前章节细纲、上一章正文定稿、当前卷 `write_brief_cache.md` 和当前卷 `chapter_plan_cache.md`。若对应文件尚未建立，标注为“口径缺口”，回退读取《故事线推进表》《总纲》和全局设定。

上一章承接必须读取上一章正文定稿最后 2-3 段 + writing_state 的“上一章结尾摘要”，核对人物站位、情绪余波、未解决动作和场景地点。

当前卷禁写清单只从 `设定/当前硬口径.md` 读取。当前卷写作设定包和口径卡片只提供正向写作口径、命名和执行提醒，不作为禁写来源；跨卷术语边界读取《设定/术语口径统一表.md》，文风硬词读取《设定/文风设定.md》。

与重写 skill 的衔接：如果 `/chapter-plan` 发现已有章节正文与当前设定源冲突，不直接原位修正文，改交给 `/rewrite-plan` 制定重写方案。

与 `/write` 的衔接：`/chapter-plan` 和 `/quality-card` 输出后，都必须同步写入当前卷 `chapter_plan_cache.md`；随后由人工按 `write-brief模板.md` 填写当前章 `/write-brief` 并写入当前卷 `write_brief_cache.md`。`/write` 必须先读取 `write_brief_cache.md`，再把 `chapter_plan_cache.md` 作为计划参考；计划缓存不得覆盖 `/write-brief` 的四个创作指标、一页结构、不可改事实和最多 3 条本章禁区。章节定稿后，由 `/finalize` 流程清理或归档已完成计划与 brief。

与复盘反推的衔接：如果当前章节正文已经存在，且用户表示“不满意”“不好看”“剧情傻”“重做”“继续加强”或要求重新执行 `/chapter-plan`，必须先执行 `/chapter-review` 的思路：从正文倒推章节问题来源，再决定是微调质量卡、重做章纲，还是交给 `/rewrite-plan`。

## 缓存写入规则

当前第一卷固定写入：

`设定/大纲内容/第一卷/chapter_plan_cache.md`

第二卷及以后写入当前卷目录下同名文件：

`设定/大纲内容/第N卷/chapter_plan_cache.md`

写入要求：

1. `/chapter-plan` 生成后，覆盖缓存中的“当前活跃章节计划”区块，保留已归档章节。
2. `/quality-card` 生成后，写入同一章节下的“当前活跃质量卡”区块，不另起新剧情。
3. 缓存必须标注章节号、章节标题、状态：`active`、生成命令和生成时间；对话回显也必须在“当前章节/当前章节元数据/复盘章节”里列出同一组元数据。
4. 如果缓存里已有同章节旧计划，直接覆盖旧的 active 区块，避免 `/write` 误读多个版本。
5. 对话里仍按下方格式输出同一份内容；不要只写文件不回显。

当前章含战斗/术式/封堵/护场/拆解/追击/危机处理，补查：`设定/神秘设定.md`、`设定/战斗设定.md`、出场角色能力发展设定；复用已出现术式吟唱时补查 `设定/术式吟唱归档.md`。

当前章涉及百城/八律/公会/古脉冲突/秘史揭露/反面边界，补查：`设定/世界观设定.md`、`设定/八律中段戏剧化设定.md`；按需补查 `设定/公会体系设定.md`（公会/远征/委托）、`设定/以太体系设定.md`（以太采集/学院设施/影子训练场）、`设定/道具武器设定.md`（道具/武器/装备机制）。

## 第一卷第7-13章专项调用口径

当前章节若落在第一卷第7-13章，`/chapter-plan` 和 `/quality-card` 不在 skill 内复制剧情细纲，必须回读 `设定/当前硬口径.md`、第一卷当前写作状态、写作设定包、口径卡片、事件链和章节目录版节奏表后，再从权威源提炼本章约束。

规划输出必须额外列出“灰区调查风险锁定”，只检查维度，不替代设定源：

1. 延迟上报/有限确认边界
2. 公开外围、警戒标识和不越线边界
3. 信息层面异常与能力揭示层级
4. 学生高光、学院接手和导师后台存在感
5. 哥伦比娅与林见星的能力/判断暴露边界
6. 证据升级是否重复、违规学生是否脸谱化

若 skill 文本、缓存或旧召回与当前权威源冲突，以当前权威源为准，并在规划里标注旧口径回潮。

## 起点追读规划口径

`/chapter-plan` 必须先判断本章有没有商业连载意义，再进入设定边界和分场。设定一致只能防崩，不能替代戏剧事件。

规划时必须优先回答：

1. 本章有没有不可替代事件，而不只是发现线索。
2. 本章有没有最强画面，能让读者记住这一章。
3. 本章有没有情绪目标：好奇、紧张、爽、发冷、心疼、期待中至少一种。
4. 本章有没有人物选择或人物弱点被击中，而不是角色只在旁边记录。
5. 本章章尾有没有让读者想立刻看下一章的追读钩子。

若本章只有“查到资料/发现异常/记录线索”，没有会改变后续局势的事件，`/chapter-plan` 必须在“本章是否只有线索没有事件”处判定为“是”，并先重做本章戏剧核心，不得直接进入分场计划。

## 章节复盘反推机制

执行 `/chapter-review`，或在已有正文基础上重跑 `/chapter-plan` 时，必须先复盘正文，不得直接替原规划找补。

复盘优先读取：

1. 当前章节正文。
2. 当前卷 `chapter_plan_cache.md` 中对应章节最近一次 `/chapter-plan` 和 `/quality-card`。
3. 当前卷写作状态、事件链、章节目录版节奏表、口径卡片和当前章节细纲。
4. 上一章结尾与下一章既定承接。

反推判断顺序：

1. 正文里真正发生了什么，不按作者意图概括，只按读者能看见的事件概括。
2. 这一章是否产生不可逆后果：关系变化、危险升级、证据升级、人物选择、规则认知变化或章尾新问题。
3. 读者情绪是否成立：好奇、紧张、爽、发冷、心疼、期待至少一种是否真实落地。
4. 主角团是否做出有效选择，而不是只观察、记录、解释或等待别人处理。
5. 章尾是否逼出下一章阅读理由，而不是只停在“还有异常”。
6. 问题来源归类：章纲设计问题、正文执行问题、设定边界过硬导致戏剧被压扁、追读钩子不足、人物高光错位。
7. 反推处理：能微调则列微调点；核心事件错误则要求重做章纲；若事件链已改变，指出需要同步的设定/大纲文件。

复盘结论必须敢于否定当前版本。不能用“铺垫”“氛围”“设定一致”掩盖本章没有事件、没有选择、没有后果的问题。

若只是执行问题，进入 /self-revise；若核心事件错误，进入 /rewrite-plan；若后续章节受影响，先重跑 /chapter-plan。

## 拆章与合章执行口径（/split-chapter）

**节拍制分流（2026-06-12 起）**：先判断属于哪种场景——

- **写作时的节拍承载调整**（尚未生成需要切割的正文文件：一个节拍拆成多章写、一章吃一个半节拍）：这**不是** `/split-chapter`，零结构同步。由 `/chapter-plan` 在"对应节拍"字段声明吃入范围，定稿时由 `/finalize` 登记映射行即可。
- **已生成正文文件或已定稿正文的结构变更**（把已有正文文件/已入库章节拆开或合并）：才走下方 `/split-chapter` 流程。且节拍制下 A5/A6/A9/A10 大幅简化：规划文件中的 ChN 是节拍编号、不随正文章号变动，通常只需"核验无需更新"；真正要改的是章↔节拍映射表、伏笔表中的实际章号和正文文件（B 层）。

`/split-chapter` 是结构性变更命令，影响章↔节拍映射、实际章号、正文文件、伏笔和缓存。**禁止在写作/检查流程中途触发**。

### 前置条件

执行前必须确认：
1. 用户已明确裁决拆分方案：原章节/正文文件 → 新章节/正文文件映射、每章事件边界和章尾钩子。
2. 已确认影响范围：章↔节拍映射、伏笔实际章号、正文文件名、缓存条目、下游正文章号偏移（如有）。
3. 当前无正在执行的 `/write`、`/finalize` 或检查命令。

若用户只说"我想拆章"但未给出方案，可协助分析建议（如事件过密点、字数预估、切割点），但**不得自行决定并执行拆分**，必须等用户确认方案后再执行。

### 执行清单

按以下清单逐项执行，每完成一项向用户报告进度并标注 ✅。

**A. 计划层同步（所有拆章场景必做）**

| 序号 | 操作 | 目标文件 | 要点 |
|------|------|---------|------|
| A1 | 更新章节结构声明 | `当前写作状态.md` 一、当前状态 | 修改版本状态、正文状态、当前可写章节、拆章说明（含日期、原→新映射、每章事件边界） |
| A2 | 更新伏笔表 | `当前写作状态.md` 四、活跃伏笔 | 更新所有受影响伏笔的"埋设章节"和"计划回收"章节号；新增伏笔补入 |
| A3 | 更新下一章任务卡 | `当前写作状态.md` 五、下一章任务卡 | 章节号、承接关系和要点更新到新章节号 |
| A4 | 更新上一章结尾摘要 | `当前写作状态.md` 二、上一章结尾摘要 | 若拆分改变了最新已完成章节，更新摘要 |
| A5 | 更新事件链章节号 | `事件链.md` | 全文替换受影响的章节号引用（明线事件链、暗线事件链、核心口径中的 Ch 编号） |
| A6 | 更新节奏表 | `章节目录版节奏表.md` | 章节表增删行、阶段比例表调整章节范围、阶段执行段落更新 |
| A7 | 更新 chapter_plan_cache | `chapter_plan_cache.md` | 原章节计划标注 superseded；为每个新章节创建 pending 占位（含事件边界和承接说明） |
| A8 | 更新 write_brief_cache | `write_brief_cache.md` | 原 active brief 归档或拆分；为新章节分别创建 brief 骨架（若即将连续写作） |
| A9 | 检查硬口径 | `当前硬口径.md` | 搜索所有受影响的章节号引用并更新；若不涉及口径变更则标注"无需更新" |
| A10 | 检查写作设定包和口径卡片 | `写作设定包.md`、`口径卡片.md` | 搜索受影响的章节号引用并更新 |

**B. 正文层操作（正文拆章 / 定稿后拆章时执行）**

| 序号 | 操作 | 要点 |
|------|------|------|
| B1 | 拆分正文文件 | 按事件边界切割正文到新的 `第N章_标题.md` 文件；确保每章有独立标题行 |
| B2 | 重命名下游正文文件 | 按人工裁决范围重命名受影响的下游已写正文文件 |
| B3 | 检查章首章尾衔接 | 新拆出的每章章首必须有承接段；章尾必须有钩子；若切割点不自然，补写过渡句（不超过 2 句） |
| B4 | 更新主角当前状态 | `当前写作状态.md` 三、主角当前状态：若拆分导致能力展示、关系登场的章节号变化，同步更新 |

**C. 入库层清理（定稿后拆章时执行，由定稿入库 skill 接管）**

C 层操作不在本 skill 执行范围内。A/B 层完成后，若涉及定稿后拆章，提示用户执行 `/finalize` 对拆分后的每章重新入库。

### 拆章后恢复

A/B 层完成后，根据当前状态提示恢复路径：
- 新章节尚未写正文：提示进入 `/chapter-plan` → `/write-brief` → `/write`
- 新章节已有正文（从拆分得到）：提示进入 `/word-check` → `/self-revise` → 正常检查 → `/finalize`
- 定稿后拆章（已完成 C 层）：直接进入下一章写作

### 拆章约束

1. 拆章是人工裁决：Agent 可建议拆章（如 `/chapter-plan` 发现事件过密），但不得自行执行。
2. 一次拆完：同一章不分多次拆分。
3. 不补新设定：拆章只重新分配已有事件，不引入新事件或新人物。
4. 章节号唯一性：完成后全项目不得存在重复或缺号章节号；立即搜索 `第\d+章` 确认无遗漏。
5. 与重写流程的关系：若拆章后某章需大幅改写，对该章走 `/rewrite-plan`，不走正常 `/write`。

### 合章操作

合章是拆章的反向操作：合并目标章节的计划、正文和缓存，下游正文章号偏移取负值（如有），伏笔表、章↔节拍映射和缓存同步更新。

## 硬约束

1. 不写正文。
2. 不输出检索过程。
3. 不反问用户。
4. 不读取旧版正文作为反例；如果用户是在重写旧章，应提示改用 `/rewrite-plan`。
5. 若状态文件和章节母版冲突，优先提示冲突，不继续生成模糊计划。
6. `/quality-card` 只能从当前状态、章节目标、事件链和最近的 `/chapter-plan` 中提炼，不新增关键剧情。
7. `/chapter-plan` 若判定“本章只有线索没有事件”，必须先重做戏剧核心；不得用分场计划包装空事件。
8. `/chapter-review` 不得把“设定一致”当作质量合格；设定一致只是底线，章节必须有事件、选择、后果和追读理由。
9. 已有正文被判定为核心事件错误时，不允许只补几句解释掩盖问题，必须交给 `/rewrite-plan` 重做。
10. 规划涉及反面时，必须提醒“反面不是所有异常的总源头”；涉及八律时，必须落到现实利益和古脉责任；涉及高位战力时，必须控制前中期展示尺度。
11. `/chapter-plan` 和 `/quality-card` 输出后，必须写入当前卷 `chapter_plan_cache.md`；下一次 `/write` 必须先读取当前章 `/write-brief` / `write_brief_cache.md`，再参考最近一次计划中的不可改事实、分场计划和质量任务卡；若计划缓存与 `/write-brief` 或权威源冲突，以 `/write-brief` 和权威源为准并先提示冲突，不继续按错误计划续写。
12. `/write` 后必须执行 `/word-check`。低于 4000 字判定为章节未完成，只能继续补写当前章；4000-4500 字判定为偏短，优先补厚；4500-6000 字才视为正常章节主体。
13. 若本章涉及主角伪装态出手或身份遮掩，必须按《设定/术语口径统一表.md》《设定/战斗设定.md》《设定/人物设定/主角能力发展设定.md》写明主动降档、输出收束、动作包装或术式外观遮掩；不得规划成体内压制力量、权柄反噬或失控风险。
