### 2.1 Master 总导演调度制

本项目的写作流程采用 **Master 总导演调度制**，不默认把所有检查命令串成固定长链。

执行原则：

1. **角色 = 判断维度**：负责判断本章需要关注什么风险。
2. **skill/命令 = 执行动作**：负责读取文件、生成正文、检查或入库。
3. **Master = 流程裁决器**：先判断章节类型和风险等级，再选择最短可用流程。

角色与命令映射：

| 角色 | 代表命令 | 主要职责 |
| ---- | -------- | -------- |
| Master 总导演 | 本文件调度规则 | 判断章节类型、风险等级、是否加检；不直接写正文 |
| 世界观总编 | `/check-setting` `/setting-term-check` `/review` | 检查世界观、术语体系、反面/以太/神秘体系是否冲突 |
| 人设监督 | `/daily-style-check` `/continuity-check` `/review` | 检查人物声轨、关系阶段、认知边界、行为是否崩人设 |
| 剧情规划师 | `/chapter-plan` `/chapter-review` `/rewrite-plan` | 判断本章事件是否成立、分场是否合理、是否需要重做 |
| 伏笔控制器 | `/continuity-check` `/chapter-plan` `/finalize` | 检查伏笔是否提前泄露、遗漏或需要定稿入库追踪 |
| 战力审计员 | `/setting-term-check` `/scene-check` `/check-setting` | 检查战力边界、术式口径、主角降档和越级暴露风险 |
| 文风润色师 | `/self-revise` `/fix-prose` `/hardword-check` | 修正文工艺、AI 腔、句式重复、硬词套路 |
| AI腔清理师 | `/hardword-check` `/self-revise` `/fix-prose` | 检查说明书腔、提示词腔、审稿腔、总结式旁白、套路句、同构句式和空泛抽象表达 |
| 情绪张力师 | `/plot-review` `/self-revise` `/daily-style-check` | 检查情绪推进、关系温度、爽点回报和章节张力 |
| 日常轻小说师 | `/daily-style-check` | 检查日常、人味、关系推进、变身适应和生活语用 |
| 神性描写师 | `/scene-check` `/setting-term-check` `/self-revise` | 检查三月权柄、神性气质、隐藏身份风险和神秘痕迹尺度 |
| 战斗演出师 | `/scene-check` `/self-revise` | 检查站位、动作线、术式画面、受击结果和战斗收口 |
| 设定检索员 | `/setting-term-check` `/check-setting`，以及 `/write` 内部检索 | 读取权威源，裁决设定冲突，不自行补设定 |
| 连载节奏编辑 | `/quality-card` `/plot-review` `/word-check` | 判断商业读感、章尾钩子、节奏密度和字数完成度 |
| 正文写手 | `/write` | 按 brief 写正文，不负责审稿和定稿 |
| 文风规则审计员 | `/check-setting` `/check-skill` | 检查文风设定文件内部一致性、规则冲突、旧口径残留、正文是否违反文风硬约束；修改文风设定后必须调用 |
| 定稿归档员 | `/finalize` `/upload-*` | 更新状态、章节摘要、伏笔表、references 和 Qdrant |
| 拆章协调员 | `/split-chapter` | 执行人工裁决的拆章/合章操作，同步全文件链章节号、伏笔引用、缓存和 Qdrant；不自行决定拆分方案，只执行已确认的拆分 |

`/logic-style-check` 暂不纳入 Master 标准调度；除非后续补齐独立执行口径，否则不作为 4.1 正常写新章节的默认步骤。

## 三、裁决优先级

```
设定文件.md（项目总纲层：不可变全局硬设定）        ← 冲突时停止并人工裁决
设定/写作规则单一来源索引.md（字段级定稿源索引）     │
设定/当前硬口径.md                          ← 日常执行最高
当前卷当前写作状态.md                        │
当前卷写作设定包.md                          │
当前卷口径卡片.md                            │
当前卷分卷大纲 / 事件链 / 节奏表 / 钩子统计        │
正文定稿                                    │
chapter_plan_cache.md（低优先级声明）         │
Qdrant 召回                                 │
references 降级参考                          ← 最低
```

若任何低优先级源与高优先级源冲突，以高优先级源为准，并标注冲突来源。卷级大纲四件套（分卷大纲/事件链/节奏表/钩子统计）之间互相冲突时，以分卷大纲为准，并同步修复其余三件（2026-06-11 补：阶段边界三版漂移的教训）。

若当前硬口径与全局不可变硬设定（主角真实身份、三月权柄、七卷月线首秀节奏）冲突，停止并提示人工裁决。

## 四、标准流程路线图

### 4.1 正常写新章节

Master 先判定章节类型、风险等级和本章主要场景，再选择流程。默认不跑全量检查。

#### A. 默认低风险流程

适用：普通校园推进章、低风险日常过渡章、无战斗、无大段设定显影、无身份暴露风险。

```
/chapter-plan → /write-brief（人工填写并写入当前卷 write_brief_cache.md）
→ /write → /word-check
→ /self-revise
→ /continuity-check
→ /hardword-check
→ /finalize
```

默认启用角色：

```text
Master 总导演
→ 剧情规划师
→ 正文写手
→ 连载节奏编辑
→ 文风润色师 / AI腔清理师
→ 人设监督 / 伏笔控制器
→ 定稿归档员
```

#### B. 条件加检

Master 根据章节内容选择加检，不把加检命令固定塞进每一章。

| 触发条件 | 加检命令 | 启用角色 |
| -------- | -------- | -------- |
| 宿舍、食堂、洗护、换衣、被照顾、被注视、室友互动、生活规则、变身适应是本章重点 | `/daily-style-check` 必跑；若命中诊断式旁白，先 `/fix-prose` 再进入 `/finalize` | 日常轻小说师、人设监督、情绪张力师 |
| 战斗、训练、术式对抗、站位调度明显 | `/scene-check` | 战斗演出师、战力审计员 |
| 设定显影、术语密集、能力解释、世界观边界明显 | `/setting-term-check` | 世界观总编、设定检索员、战力审计员 |
| 主角身份、三月权柄、神性痕迹、隐藏风险明显 | `/scene-check` + `/setting-term-check` | 神性描写师、世界观总编、伏笔控制器 |
| 节奏存疑、不好看、钩子弱、商业回报不足 | `/plot-review` | 连载节奏编辑、情绪张力师、剧情规划师 |
| 已写正文与计划冲突、事件结果可能需要改 | `/chapter-review` 或 `/rewrite-plan` | 剧情规划师、连载节奏编辑 |
| 修改了文风设定、写作规则或审稿skill | `/check-setting` + `/check-skill` | 文风规则审计员、世界观总编 |
| 刚执行过拆章/合章，首次写新章节号 | `/continuity-check` 必跑；确认全文件链章节号同步无遗漏 | 拆章协调员、伏笔控制器、人设监督 |

战斗章、设定章、身份风险章：每写完一场可先跑 `/scene-check`，不必等整章完成。

#### C. 高风险全检流程

适用：战斗章、设定章、身份风险章、重写后章节、连续性复杂章节，或用户明确要求“全量定稿前检查”。

```text
/chapter-plan → /quality-card → /write-brief（人工填写并写入当前卷 write_brief_cache.md）
→ /write → /word-check
→ /self-revise
→ /daily-style-check（若含日常/关系/变身适应）
→ /scene-check
→ /continuity-check
→ /setting-term-check
→ /hardword-check
→ /finalize
```

对于节奏存疑或已写完但感觉差点意思的章节，可在定稿前插入：

```
/plot-review → （根据建议）/fix-prose 或 /rewrite-plan → /finalize
```

### 4.2 重写跑偏章节

```
/rewrite-plan → /rewrite-next → /rewrite-check → /rewrite-fix
（重复 next/check/fix 直到整章完成）
→ /rewrite-final-check → /finalize
```

### 4.3 定稿后修改

| 修改级别 | 触发条件               | 流程                                                   |
| ---- | ------------------ | ---------------------------------------------------- |
| 小修   | 只改错字、AI腔、虚感比喻      | `/hardword-check` → `/refinalize-text-only`          |
| 中修   | 改动作、反应、对白，不改事件结果   | `/scene-check` + `/continuity-check` → `/refinalize` |
| 大修   | 改事件结果、胜负、身份暴露、关系走向 | `/rewrite-plan` → 完整重写流程 → `/refinalize-major`       |

### 4.4 拆章与合章调整

拆章/合章是结构性变更，影响全文件链的章节号引用。必须在动笔前或定稿后执行，**禁止在写作/检查流程中途拆章**。

触发命令：`/split-chapter`（人工裁决后触发，Master 执行同步操作）。

#### 适用场景

| 场景 | 触发条件 | 复杂度 |
|------|---------|--------|
| 计划拆章 | 单章 chapter_plan 事件过密或预估字数超 5000 字，尚未写正文 | 低——只改计划层 |
| 正文拆章 | 已写完正文但未定稿，发现单章过长或事件可分 | 中——需拆分正文 + 改计划层 |
| 定稿后拆章 | 已定稿入库的章节需要拆分 | 高——需拆分正文 + 改计划层 + 清理 Qdrant + 重新定稿 |
| 合章 | 两章内容过短或事件关联性太强，合并为一章 | 与拆章反向操作，同复杂度等级 |

#### 前置条件

1. **人工裁决拆分方案**：明确原章节号 → 新章节号映射（如"原 Ch4 → Ch4/Ch5/Ch6"）、每章事件边界和章尾钩子。
2. **确认下游影响范围**：原章节号之后的所有章节号偏移量（如 Ch4 拆为 3 章 → 后续章节号 +2）。
3. **暂停当前写作流程**：拆章完成前不得执行 `/write`、`/finalize` 或任何检查命令。

#### 执行清单

拆章协调员按以下清单逐项执行，每完成一项标注 ✅。

**A. 计划层同步（所有拆章场景必做）**

| 序号 | 操作 | 目标文件 | 要点 |
|------|------|---------|------|
| A1 | 更新章节结构声明 | `当前写作状态.md` 一、当前状态 | 修改版本状态、正文状态、当前可写章节、拆章说明（含日期、原章节号→新章节号映射、每章事件边界） |
| A2 | 更新伏笔表 | `当前写作状态.md` 四、活跃伏笔 | 所有受影响的伏笔条目：更新"埋设章节"和"计划回收"中的章节号；新增伏笔如有则补入 |
| A3 | 更新下一章任务卡 | `当前写作状态.md` 五、下一章任务卡 | 章节号、承接关系和要点全部更新到新章节号 |
| A4 | 更新上一章结尾摘要 | `当前写作状态.md` 二、上一章结尾摘要 | 若拆分改变了最新已完成章节，更新摘要 |
| A5 | 更新事件链章节号 | `事件链.md` | 全文替换受影响的章节号引用（明线事件链、暗线事件链、核心口径中的 Ch 编号） |
| A6 | 更新节奏表 | `章节目录版节奏表.md` | 章节表增删行、阶段比例表调整章节范围、阶段执行段落更新 |
| A7 | 更新 chapter_plan_cache | `chapter_plan_cache.md` | 原章节计划标注 superseded；为每个新章节创建 pending 占位（含事件边界和承接说明） |
| A8 | 更新 write_brief_cache | `write_brief_cache.md` | 原 active brief 归档或拆分；为新章节分别创建 brief 骨架（若即将连续写作） |
| A9 | 检查硬口径 | `当前硬口径.md` | 搜索所有受影响的章节号引用并更新；若不涉及口径变更则跳过 |
| A10 | 检查写作设定包和口径卡片 | `写作设定包.md`、`口径卡片.md` | 搜索受影响的章节号引用并更新 |

**B. 正文层操作（正文拆章 / 定稿后拆章时执行）**

| 序号 | 操作 | 要点 |
|------|------|------|
| B1 | 拆分正文文件 | 按事件边界切割正文到新的 `第N章_标题.md` 文件；确保每章有独立标题行 |
| B2 | 重命名下游正文文件 | 原章节号之后的已写正文文件按新编号重命名 |
| B3 | 检查章首章尾衔接 | 新拆出的每章章首必须有承接段；章尾必须有钩子；若切割点不自然，补写过渡句（不超过 2 句） |
| B4 | 更新主角当前状态 | `当前写作状态.md` 三、主角当前状态：若拆分导致能力展示、关系登场等记录的章节号变化，同步更新 |

**C. 入库层清理（定稿后拆章时执行）**

| 序号 | 操作 | 要点 |
|------|------|------|
| C1 | 清理 Qdrant 旧数据 | 删除原章节号在 chapter 集合中的旧向量条目 |
| C2 | 重新定稿入库 | 对拆分后的每章重新执行 `/finalize` 入库流程 |
| C3 | 同步 references | 若 `D:/Users/yueguang/NovelFlow/references` 有旧文件，删除旧文件并同步新文件 |

#### 拆章后恢复流程

拆章完成后，根据当前状态选择恢复路径：

```
拆章完成
→ 若新章节尚未写正文：/chapter-plan → /write-brief → /write → 正常流程
→ 若新章节已有正文（从拆分得到）：/word-check → /self-revise → 正常检查流程 → /finalize
→ 若新章节已定稿（定稿后拆章）：已完成 C 步骤，直接进入下一章写作
```

#### 合章操作

合章是拆章的反向操作，执行清单同理但方向相反：
- 合并目标章节的计划、正文和缓存
- 下游章节号偏移量取负值（如两章合一 → 后续章节号 -1）
- 伏笔表、事件链、节奏表同样全文更新

#### 注意事项

1. **拆章是人工裁决**：Agent 可以建议拆章（如在 `/chapter-plan` 发现事件过密时），但不得自行执行拆分，必须等人工确认方案。
2. **一次拆完**：同一章不要分多次拆分；若预估还需再拆，先完成全部拆分再进入写作。
3. **不补新设定**：拆章只重新分配已有事件，不引入原计划中没有的新事件或新人物。
4. **章节号唯一性**：拆章完成后，全项目不得存在重复章节号或缺号；立即搜索 `第\d+章` 确认无遗漏。
5. **与重写流程的关系**：若拆章后某章需要大幅改写（不只是切割），对该章走 `/rewrite-plan` 流程，不走正常 `/write`。

## 五、写前必查五件事

每次 `/write` 前必须先人工填写 `/write-brief` 并写入当前卷 `write_brief_cache.md`，第一卷路径为 `设定/大纲内容/第一卷/write_brief_cache.md`。随后确认：

1. 当前章节号和上一章结尾是否正确
2. 本章写作目标、四个创作指标和最多 3 条本章禁区是否明确
3. 如果涉及百城/八律/公会/秘史/高位战力，是否已在计划里写明世界观边界
4. 缓存中的分场计划是否与当前卷状态冲突——冲突时以状态文件为准
5. 近期是否执行过拆章/合章——若有，确认当前写作状态、事件链、节奏表、缓存文件的章节号已全部同步，且当前章节号在状态文件中有对应条目

## 七、规则单一来源与镜像壳维护（2026-06-11 架构改版）

skill 规则正文的**唯一来源是 `.trae/rules/`**。其余平台目录全部是薄壳/路由层，不保存规则正文：

- `.trae/rules/`（**规则正文唯一来源**，8 个 skill 源文件）
- `.claude/skills/`（薄壳：frontmatter + 一句"读取 `.trae/rules/` 对应源文件并执行"）
- `.github/skills/`（薄壳：同上）
- `.windsurf/workflows/`（slash 路由层：只做命令路由和执行步骤，引用 `.trae/rules/`）

维护原则：

1. **改规则只改 `.trae/rules/` 源文件**。`.claude`/`.github` 壳文件正文永远不需要改。
2. 壳文件的 frontmatter（name、description、version、trigger、tools）以 trae 源文件 frontmatter 为准；**只有当 trigger、description 或 tools 声明变化时**，才需要把源文件 frontmatter 重新同步到两个壳（机械复制，不做内容判断）。
3. 所有 skill 文件（源与壳）必须为无 BOM 的 UTF-8——文件开头若带 BOM（`EF BB BF`），frontmatter 解析会失败、skill 元数据失效（2026-06-11 审稿 skill 故障的根因）。
4. Windsurf workflow 文件名必须与 slash command 去掉 `/` 后一致，例如 `/review` 对应 `.windsurf/workflows/review.md`；workflow 不复制长篇规则，只引用 `.trae/rules/` 当前源文件。
5. skill 源文件不保存章节细纲级剧情快照；涉及当前剧情、旧训楼、灰区调查、敌方单位、月线展示等会变化内容时，只写"读取权威源 + 检查维度"。
6. `/write` 和 `/word-check` 归续写 skill；`/finalize` 和 `/upload-*` 只归定稿入库 skill，禁止两个 skill 同时接管。
7. 每次大改设定、章节结构或工作流后，建议执行 `/check-skill`：检查壳文件指向的源文件存在、frontmatter 与源一致、workflow 覆盖、路由冲突、旧口径回潮和设定备份化问题；**不再做三目录正文逐行比对**（壳无正文可比）。
8. 旧 skill、旧入口、废弃规则不作占位保留；确认新入口承接后，直接删除旧文件和旧目录，避免后续 agent 误加载。
